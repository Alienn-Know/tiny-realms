## Pre-training vs Fine-tuning: Откуда берётся "ум"

### Pre-training = построить мозг
Модель учится понимать мир: логика, язык, факты, рассуждения. Это building blocks интеллекта.

```
Base model после pre-training:
├── Понимает что такое "сортировка" и "алгоритм"
├── Знает синтаксис Rust
├── Видит связь между разными концепциями
└── Но: не знает что от неё хотят
```

### Fine-tuning = обучить профессии
Направляет существующий "ум" в конкретное русло. Не добавляет "ума", добавляет "полезность".

```
После fine-tuning:
├── "напиши функцию сортировки" → работающий код
├── "объясни ошибку" → понятное объяснение
└── Но базовый "ум" — тот же что был после pre-training
```

### Ключевой вывод
Fine-tuning НЕ делает модель умнее — учит её БЫТЬ ПОЛЕЗНОЙ.

| Что хочешь | Нужен pre-training | Нужен fine-tuning |
|-------------|-------------------|-------------------|
| Модель умнее понимает мир | ✅ Да | ❌ Нет |
| Модель лучше следует инструкциям | ❌ Нет | ✅ Да |
| Модель специализируется на задаче | ❌ Нет | ✅ Да |

## Специализированная модель vs Универсальная (Rust example)

### Сравнение для Rust coding

| Задача | GPT-5.5 (универсальная) | Rust-Super-30B (спец.) |
|--------|------------------------|----------------------|
| Простой код | 90% | 95% |
| Средний код | 75% | 90% |
| Сложный код | 55% | 88% |
| Отладка borrow | 60% | 92% |
| Отладка unsafe | 40% | 85% |
| Проектирование | 65% | 92% |
| Оптимизации | 50% | 90% |

**Преимущество специализации: 15-45% лучше для своей ниши**

### Когда универсальная всё ещё лучше
- Задача затрагивает другие языки (Python, Go, Java)
- Нужен full-stack (backend + frontend)
- Критично — нельзя ошибаться (production)
- Логика предметной области (финансы, медицина)

## Паттерн именования моделей

```
Qwen 3          → Большое обновление архитектуры
Qwen 2.5         → Итерация того же поколения
Qwen 2           → Предыдущее поколение

Qwen3-30B        → Базовый инструктаж (base)
Qwen3-30B-Instruct → Настроенный следовать инструкциям
Qwen3-30B-Chat   → Настроенный для диалогов
Qwen3-30B-A3B    → MoE версия (3B active из 30B total)
```

### Суффиксы

| Суффикс | Значение | Когда брать |
|---------|----------|-------------|
| -Base или без суффикса | Не обучена следовать инструкциям | Только для fine-tune |
| -Instruct | Обучена следовать инструкциям | Для generic использования |
| -Chat | Оптимизирована для диалогов | Если нужен чат-интерфейс |
| -A3B или -AxxB | MoE архитектура (active params) | Для эффективности |

## Большие версии vs Итерации

### Qwen 2 → Qwen 2.5 (итерация)
```
Изменения:
├── Более качественные данные pre-training
├── Лучше понимает современные вещи
├── Улучшен reasoning
└── Но архитектура та же

Когда обновляться:
✅ Если модель старая (Qwen 1 → Qwen 2)
⚠️ Если уже есть Qwen 2.5 — Qwen 2.6 не критично
```

### Qwen 2 → Qwen 3 (большое обновление)
```
Изменения:
├── Новая архитектура (Hybrid DeltaNet, новый attention)
├── Новый tokenizer
├── Больше parameters / better efficiency
├── Значительно улучшен reasoning
└── Может быть совсем другой характер ответов

Когда обновляться:
✅ Всегда стоит посмотреть — часто большой скачок
✅ Особенно если Qwen 2 был base для fine-tune
```

## Критерии выбора базовой модели

### 1. Архитектура

```
Dense (обычная)              MoE (mixture of experts)
├── Все params активируются  ├── 100B total, но 3-10B active
├── Предсказуемость          ├── Дешевле inference
└── Дорого                   └── Непредсказуемость
```

**Когда Dense лучше:**
- Нужна стабильность
- Задачи требуют всех capabilities модели
- Меньше данных для fine-tuning

**Когда MoE лучше:**
- Хочешь больше "ума" за те же деньги
- Limited VRAM (2x3090)

### 2. Размер vs Возможности

| Модель | VRAM | Для чего | Ограничения |
|--------|------|----------|-------------|
| 7B | 14-16GB | Простые задачи | Слабая для сложного |
| 14B | 28-32GB | Средние задачи | 2x3090 хватит |
| 30B dense | 60GB+ | Сложные задачи | Нужен H100/A100 |
| 30B MoE | 18-36GB | Сложные задачи | 2x3090 возможно |
| 100B+ MoE | 200GB+ | Frontier | Невозможно локально |

### 3. Качество pre-training данных
Самый важный критерий. Косвенно оценить через:
- Benchmarks на reasoning (GPQA, AIME)
- Общее качество ответов
- Отсутствие hallucination
- Знание современных вещей (2024-2026)

### 4. Context window
Правило: Context должен быть больше чем типичная задача × 2-3

### 5. Лицензия
```
Apache 2.0 / MIT     → Можно всё, коммерция ✅
GPL                 → Code должен быть открыт ❌
Custom              → Ограничения читать осторожно ❌
```

### 6. Community & Support
- Готовые LoRA adapters
- vLLM/Ollama поддержка
- Active community

## Формула выбора базовой модели

```
Базовая модель = (достаточно умная) 
              + (запускается на твоём железе) 
              + (permissive license) 
              + (активная community)

Шаг 1: Определи максимальный размер VRAM
Шаг 2: Выбери архитектуру (MoE если мало VRAM)
Шаг 3: Проверь license (Apache/MIT обязательно)
Шаг 4: Посмотри benchmarks на reasoning задачах
Шаг 5: Проверь community support (есть ли vLLM/Ollama)
Шаг 6: Проверь context window (должен быть больше задачи)
```

## Стоит ли всегда брать latest версию?

```
✅ ДА, если:
├── Новая версия значительно лучше (Qwen 2 → Qwen 3)
├── Используешь готовую модель (не fine-tune)
├── Нужны свежие знания
└── Есть hardware для новой модели

❌ НЕТ (подожди), если:
├── Уже вложился в fine-tune текущей версии
├── Нужна стабильность для production
├── Community ещё не нашла баги
├── Текущая версия полностью покрывает задачи
```
